Git & GitHub Commands Cheat Sheet

A comprehensive guide to Git and GitHub for version control

Introduction to Git and GitHub

Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency.

GitHub is a web-based hosting service for Git repositories that provides collaboration features, code review, and issue tracking.

Key difference: Git is the tool that tracks changes in your code, while GitHub is a website where you host your repositories.

Getting Started

Configuring Git

Before you start using Git, you need to configure your identity:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Verify your configuration:

git config --list

Using --global sets this configuration for all your Git repositories. To set it just for a specific repository, navigate to that repository and omit --global.

Creating a Repository

Initialize a new Git repository in your project folder:

git init

This creates a hidden .git directory that contains the Git database and configuration.

Cloning an Existing Repository

Create a copy of an existing Git repository:

git clone "https://github.com/username/repository.git"

Clone with SSH (requires SSH key setup):

git clone "git@github.com:username/repository.git"

HTTPS is easier to set up but requires entering your credentials each time. SSH is more secure and doesn't require password entry after setup.

Basic Git Commands

Checking Status

View the status of your working directory:

git status

This command shows:

  • Current branch
  • Changed files that are not staged
  • Changes that are staged but not committed
  • Files that are not tracked by Git

Adding Files

Add files to the staging area:

git add filename.txt

Add multiple specific files:

git add file1.txt file2.txt

Add all changed files:

git add .

The staging area is a middle ground between your working directory and the repository. It allows you to prepare a set of changes before committing them.

Committing Changes

Record the changes in the repository:

git commit -m "Add a descriptive commit message here"

Add and commit in one step (for already tracked files):

git commit -am "Updated existing files"

Write meaningful commit messages that explain why a change was made, not just what was changed.

Pushing Changes

Upload local repository content to a remote repository:

git push origin main

Set upstream and push (first time):

git push -u origin main

After setting upstream, you can simply use:

git push

Working with Remotes

Add a remote repository:

git remote add origin https://github.com/username/repository.git

View remote repositories:

git remote -v

Remove a remote:

git remote remove origin

Pulling Changes

Fetch and merge changes from the remote repository:

git pull origin main

This is equivalent to:

git fetch origin
git merge origin/main

git fetch downloads changes without merging them, while git pull fetches and merges in one step.

Working with Branches

Branch Management

List all branches:

git branch

Create a new branch:

git branch feature-name

Create and switch to new branch in one command:

git checkout -b feature-name

Switch to an existing branch:

git checkout branch-name

Delete a branch (after merging):

git branch -d branch-name

Force delete a branch (even if not merged):

git branch -D branch-name

Rename current branch:

git branch -M main

Pushing a New Branch

Push a branch to remote and set up tracking:

git push --set-upstream origin feature-name

Short form:

git push -u origin feature-name

Comparing and Merging

View differences between branches:

git diff branch1 branch2

Merge a branch into your current branch:

git merge branch-name

Before merging, make sure to commit or stash any changes in your working directory to avoid conflicts.

Handling Merge Conflicts

Understanding Merge Conflicts

Merge conflicts occur when Git can't automatically resolve differences between two commits. This typically happens when two branches have made changes to the same part of the same file.

Git marks the conflicted areas in files with special markers:

<<<<<<< HEAD
Your changes
=======
Their changes
>>>>>>> branch-name

Resolving Conflicts

Steps to resolve merge conflicts:

  1. Run git status to see which files have conflicts
  2. Open the conflicted files in your editor and look for the conflict markers
  3. Edit the files to resolve the conflicts (remove the markers and keep the desired code)
  4. Add the resolved files: git add <filename>
  5. Continue with the merge: git merge --continue or create a new commit

Use visual merge tools like VS Code, GitKraken, or SourceTree to make resolving conflicts easier.

Abort a Merge

If you want to cancel a merge that has conflicts:

git merge --abort

Fixing Mistakes

Unstaging Files

Remove files from the staging area:

git reset filename

Remove all files from staging:

git reset

Undoing Commits

Undo the most recent commit, keeping changes staged:

git reset --soft HEAD~1

Undo the most recent commit, unstaging changes:

git reset HEAD~1

Undo multiple commits (e.g., last 3):

git reset HEAD~3

Undo commits and discard all changes (CAUTION - data loss):

git reset --hard HEAD~1

--hard flag will permanently discard changes. Use with caution!

Reverting Commits

Create a new commit that undoes a previous commit:

git revert commit-hash

Find commit hashes with:

git log

revert is safer than reset for shared branches because it doesn't rewrite history.

Recovering Lost Changes

View reference logs (reflog) to find lost commits:

git reflog

Restore a branch to a specific reflog entry:

git reset --hard HEAD@{n}

Where n is the entry number from reflog.

Advanced Git Concepts

Stashing Changes

Temporarily store modified files:

git stash

List stashed changes:

git stash list

Apply the most recent stash:

git stash apply

Apply a specific stash:

git stash apply stash@{n}

Apply and remove the stash:

git stash pop

Discard a stash:

git stash drop stash@{n}

Viewing History

View commit history:

git log

View a compact history:

git log --oneline

View history with branch graph:

git log --oneline --graph --all

View changes in a specific commit:

git show commit-hash

GitHub Concepts

Forking Repositories

A fork is a copy of a repository that allows you to freely experiment without affecting the original project.

Steps to fork:

  1. Click the "Fork" button on a GitHub repository page
  2. Clone your forked repository to your local machine
  3. Add the original repository as an "upstream" remote to keep your fork in sync
git remote add upstream https://github.com/original-owner/original-repository.git

Sync your fork with the original repository:

git fetch upstream
git checkout main
git merge upstream/main

Pull Requests

Pull requests let you tell others about changes you've pushed to a branch in a repository on GitHub.

Workflow for creating a pull request:

  1. Fork the repository (if it's not yours)
  2. Create a branch for your changes
  3. Make and commit your changes
  4. Push the branch to GitHub
  5. Go to the repository on GitHub and click "Compare & pull request"
  6. Fill in the PR description and create the pull request

Git Best Practices

Commit Guidelines

  • Make small, focused commits that do one thing
  • Write clear commit messages (start with a verb, keep under 50 characters)
  • Group related changes in a single commit
  • Commit often to avoid losing work

Consider using conventional commits format: type(scope): description
Example: feat(auth): add login button to navbar

Branching Strategy

Common branching models:

  • Git Flow: main, develop, feature/, release/, hotfix/ branches
  • GitHub Flow: main branch with feature branches
  • Trunk-Based Development: Small changes directly to main or short-lived feature branches

Choose a branching strategy that works for your team size and release schedule.

Pull Request Tips

  • Keep PRs small and focused on a single issue
  • Include a clear description of what changes were made and why
  • Reference related issues (#123)
  • Add screenshots for UI changes
  • Request reviews from appropriate team members
